# Technology Stack

**Project:** lz-nx.rlm -- RLM-powered Claude Code plugin for Nx JS/TS workspaces
**Researched:** 2026-03-03
**Mode:** Ecosystem research (stack dimension)

## Recommended Stack

### Runtime

| Technology | Version | Purpose | Why | Confidence |
|------------|---------|---------|-----|------------|
| Node.js | 24.x LTS (Krypton) | Plugin runtime | Active LTS through April 2028. Required minimum for Nx 21+ (20.19+). The `vm.createContext()` improvements in 24.x (`vm.constants.DONT_CONTEXTIFY`, `Error.isError()` cross-realm) directly benefit our REPL sandbox. Node 20.x EOL is April 2026 -- too close. | HIGH |

### REPL Sandbox

| Technology | Version | Purpose | Why | Confidence |
|------------|---------|---------|-----|------------|
| `node:vm` (built-in) | N/A | REPL code execution sandbox | Use because: (1) zero dependency -- ships with Node.js, (2) <5ms startup vs 100-500ms for subprocess, (3) proven in both hampton-io/RLM v0.3.0 and code-rabi/rllm v1.1.0, (4) `codeGeneration: { strings: false, wasm: false }` blocks eval attacks, (5) our threat model is LLM-generated code (not adversarial untrusted user code), so vm-level isolation is sufficient. **NOT** `isolated-vm` -- see Alternatives below. | HIGH |

**Critical security note:** The `node:vm` module is NOT a security sandbox against adversarial code. It prevents accidental escape but can be trivially escaped via prototype chain (`this.constructor.constructor('...')`) by a determined attacker. This is acceptable for our use case because:

1. The code is generated by Claude (Sonnet/Haiku), not by untrusted users
2. We restrict the sandbox globals (no `process`, `require`, `child_process`)
3. We disable `codeGeneration.strings` and `codeGeneration.wasm`
4. The plugin already runs in the user's local machine context -- there is no privilege escalation

If the plugin ever executes user-provided code (not LLM-generated), upgrade to `isolated-vm`.

## Alternatives Considered

| Category | Recommended | Alternative | Why Not |
|----------|-------------|-------------|---------|
| REPL sandbox | `node:vm` | `isolated-vm` 6.0.2 | `isolated-vm` requires native compilation (C++ addon). Fails on arm64 Windows without Visual Studio build tools. Plugin must work on macOS/Linux/Windows with only Node.js LTS. Our threat model (LLM-generated code) does not require process-level isolation. |
| REPL sandbox | `node:vm` | Subprocess REPL (Bun/Node child_process) | Higher latency per iteration (IPC overhead). opencode-rlm uses this pattern but reports 50-100ms per round-trip vs <5ms for vm. In a 20-iteration REPL loop, that is 1-2 seconds of pure overhead. |
| REPL sandbox | `node:vm` | `vm2` / `@n8n/vm2` | Deprecated. 8 critical CVEs in 2023. Revived October 2025, new critical CVE January 2026 (CVSS 9.8). Architecturally unfixable -- Node.js intercepts calls from the sandbox, preventing proper proxy wrapping. Do not use. |
| REPL sandbox | `node:vm` | QuickJS via WASM | Strongest isolation (JS interpreter in WASM sandbox) but massive complexity overhead. No native async/await. Would require reimplementing all REPL globals as WASM-accessible functions. Overkill for our threat model. |
| File globbing | `node:fs/promises` glob | `fast-glob` 3.3.3 | `fast-glob` is battle-tested (9,654 dependents) but adds a dependency. Since we target Node.js 24 (which has stable `fs.glob` since 22.x), the built-in is sufficient. Fall back to `fast-glob` only if `fs.glob` proves problematic on Windows. |
| File globbing | `node:fs/promises` glob | `glob` 11.x (npm) | Same reasoning. Built-in preferred over third-party when feature parity exists. |
| Workspace index | JSON file | SQLite via `better-sqlite3` | SQLite adds a native module dependency, complicating cross-platform support. JSON index is sufficient for v1 (50-100KB for 537 projects). SQLite becomes valuable only at >1000 projects or if query patterns require joins/indices. Defer. |
| Workspace index | JSON file | Nx MCP server tools | The Nx MCP server provides `nx_workspace` and `nx_project_details` tools, but (1) each MCP call costs a tool-use round-trip with tokens, (2) our REPL needs the data as an in-memory variable, not a tool call, (3) the Nx MCP is designed for AI agent consumption, not programmatic REPL access. |
| Test runner | `node:test` | Vitest 4.x | Vitest is excellent but adds a build/bundler dependency chain (Vite). For a plugin with <20 test files and no compilation step, built-in `node:test` is simpler. |
| Test runner | `node:test` | Jest 30.x | Jest's ESM support is still experimental. Our scripts are ESM (`.mjs`). `node:test` handles ESM natively. |
| Plugin integration | Native plugin system | MCP server | 4 of 8 existing RLM plugins use MCP. However, for our use case, the plugin system's skills + agents + hooks provide tighter integration: (1) skills auto-invoke based on context, (2) agents support model routing natively, (3) hooks fire on Claude Code events. MCP would add a server process and lose these affordances. |

## Zero External Dependencies Strategy

The plugin ships with **zero npm dependencies**. Every component uses Node.js built-in modules:

| Need | Built-in Solution |
|------|-------------------|
| REPL sandbox | `node:vm` |
| File reading | `node:fs` |
| File globbing | `node:fs/promises` glob (Node.js 22+) |
| Child processes | `node:child_process` |
| Path manipulation | `node:path` |
| JSON parsing | `JSON.parse()` / `JSON.stringify()` |
| Testing | `node:test` + `node:assert` |
| Process timing | `node:perf_hooks` |

**Why zero dependencies:**

1. **Cross-platform reliability** -- no native modules that fail on arm64 Windows, no QEMU emulation issues
2. **No install step** -- plugin works immediately after `claude plugin install`
3. **No supply chain risk** -- critical for a security-adjacent tool (sandbox execution)
4. **Simplicity** -- plugin is a collection of `.mjs` scripts and `.md` files

The only external binary required is `git` (for `git grep` in the `search()` REPL global) and `nx` (via `npx nx` in the workspace being analyzed). Both are assumed present in the target environment.

## Script Language Decision: JavaScript (.mjs), Not TypeScript

All plugin scripts are plain JavaScript with ESM syntax (`.mjs` files). Not TypeScript.

**Why:**

1. **No build step** -- scripts run directly via `node scripts/foo.mjs`. No `tsc`, no bundler, no `tsconfig.json`.
2. **Plugin convention** -- Claude Code plugins execute scripts directly. A compilation step adds fragility (stale builds, missing source maps).
3. **Cross-platform simplicity** -- no TypeScript compiler version mismatch issues.
4. **JSDoc for types** -- use `@param`, `@returns`, `@typedef` JSDoc annotations for IDE support without a compile step. TypeScript's `--checkJs` can validate JSDoc types in CI if desired.

If the plugin grows complex enough to warrant TypeScript, extract a separate `src/` -> `dist/` build pipeline. For v1 with ~10 scripts at ~100-300 lines each, `.mjs` is the pragmatic choice.

## Installation

No `npm install` required. The plugin is a directory of scripts and markdown files.

```bash
# Install the plugin
claude plugin install lz-nx.rlm@<marketplace>

# Or for development, use --plugin-dir
claude --plugin-dir ./plugins/lz-nx.rlm
```

## Model Routing via Subagent System

| Agent | Model | Claude Code Config | Rationale |
|-------|-------|-------------------|-----------|
| `repl-executor` | `sonnet` | `model: sonnet` in frontmatter | Code generation for REPL navigation requires moderate reasoning. Sonnet balances capability and cost. |
| `haiku-searcher` | `haiku` | `model: haiku` in frontmatter | Mechanical search tasks. Haiku 4.5 matches Sonnet 4 on coding benchmarks at 1/3 the cost. |
| Main conversation | `inherit` | User's chosen model | Plugin does not override the user's model choice for the main thread. |

Subagent model field accepts: `"sonnet"`, `"opus"`, `"haiku"`, or `"inherit"` (default). These are aliases that map to the current model versions (Sonnet 4.6, Opus 4.6, Haiku 4.5 as of March 2026).

## New Findings Since Existing Research (March 2026)

### Claude Code Plugin System Updates

1. **Skills and commands unified** -- Files in `commands/` still work but `skills/` is now recommended. Skills can be auto-invoked by Claude based on conversation context.
2. **Three hook types** -- `command` (shell), `prompt` (LLM evaluation), `agent` (agentic verifier with tools). The `agent` hook type is new and useful for complex validation.
3. **New hook events** -- `SubagentStart`, `SubagentStop`, `PostToolUseFailure`, `PermissionRequest`, `TeammateIdle`, `TaskCompleted` added since early research.
4. **Subagent `memory` field** -- Persistent memory scoped to `user`, `project`, or `local`. Enables cross-session learning for the REPL executor.
5. **Subagent `background` field** -- Run subagents concurrently with `background: true`. Useful for parallel exploration.
6. **Subagent `isolation: worktree`** -- Run in a git worktree for repository isolation.
7. **Subagent `skills` preloading** -- Inject skill content into subagent context at startup.
8. **HTTP hooks** -- POST JSON to a URL instead of running a shell command.
9. **LSP server support** -- Plugins can bundle Language Server Protocol servers.

### Node.js 24 vm Module Improvements

1. **`vm.constants.DONT_CONTEXTIFY`** -- Creates contexts without contextifying quirks. The global object behaves more naturally (can be frozen). Consider using for our sandbox.
2. **`Error.isError()` cross-realm** -- Properly detects Error instances across vm contexts. Fixes the classic `instanceof Error` cross-realm bug. Useful for error handling in REPL output.
3. **V8 13.6 engine** -- Full ECMAScript 2025 support in sandboxed code.

### hampton-io/RLM Updates

The hampton-io/RLM project (v0.3.0) now includes:
- MCP server integration for Claude Code
- Multi-provider support (OpenAI, Anthropic, Google)
- Extended thinking support for Claude 4.5+
- 25+ model benchmarks

Our plugin does NOT use hampton-io/RLM as a library -- we implement our own REPL because:
1. We need Nx-specific globals (`workspace`, `projects`, `deps()`, `nx()`)
2. We route through Claude Code's native subagent system, not external API calls
3. Our sandbox needs file system and child process access (hampton-io is context-string only)

But their `VMSandbox` implementation (vm-sandbox.ts) is a validated reference for:
- `const/let` -> `globalThis` transformation for variable persistence
- Async IIFE wrapping for `await` support
- Promise-based `llm_query` callback pattern
- `Object.prototype.toString` patch to prevent `[object Object]` corruption

### code-rabi/rllm Updates

The code-rabi/rllm project (v1.1.0) adds:
- Zod schema support for typed contexts
- Real-time event streaming API
- `generateObject()` for structured output

Relevant pattern: Zod schema support lets the LLM understand context structure. We achieve similar with the workspace index's typed structure.
