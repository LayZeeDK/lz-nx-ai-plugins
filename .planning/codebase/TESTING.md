# Testing Patterns

**Analysis Date:** 2026-03-03

## Status Note

This repository is in the planning phase. No test files exist yet. All patterns
below are derived from: (1) the stack decision in `.planning/research/STACK.md`
(which explicitly selects `node:test` as the test runner), (2) `nx.json` (which
configures `@nx/vitest:test` as the Nx test target), and (3) `AGENTS.md`
(which defines verification requirements for plugin changes). Two testing
approaches are relevant to different parts of the project -- see below.

## Test Framework: Two Contexts

### Context 1: Plugin Scripts (Node.js built-in test runner)

Plugin scripts under `plugins/lz-nx.rlm/scripts/` are plain `.mjs` files with
zero dependencies. The test runner for these is `node:test` (built-in,
Node.js 24 LTS).

**Runner:** `node:test` (built-in, Node.js 24 LTS)
**Assertion Library:** `node:assert` (strict mode)
**Config:** No config file needed -- runs directly via `node --test`

**Run Commands:**
```bash
# Run all tests for a plugin script directory
node --test plugins/lz-nx.rlm/scripts/**/*.test.mjs

# Run a single test file
node --test plugins/lz-nx.rlm/scripts/workspace-indexer.test.mjs

# Run with coverage (Node.js 24)
node --test --experimental-test-coverage plugins/lz-nx.rlm/scripts/**/*.test.mjs

# Run via Nx (once Nx projects are configured)
pnpm nx test lz-nx.rlm
```

### Context 2: Nx-Managed Library Projects (Vitest)

For any Nx library projects added to the workspace, the configured test target
is Vitest (via `@nx/vitest` plugin). This is the default for new Nx projects
scaffolded with `@nx/js` or `@nx/node`.

**Runner:** Vitest 4.x
**Config:** `vitest.config.ts` per project (auto-generated by `@nx/vitest`)
**Nx plugin:** `@nx/vite/plugin` (configured in `nx.json`)

**Run Commands:**
```bash
# Run tests for a specific Nx project
pnpm nx test <project-name>

# Run all affected tests
pnpm nx affected --target=test

# Run with coverage
pnpm nx test <project-name> --coverage

# Run in watch mode
pnpm nx test <project-name> --watch
```

**For the `lz-nx.rlm` plugin specifically, use `node:test`** (not Vitest) because
the plugin scripts have zero dependencies and run without a build step.
See `.planning/research/STACK.md` rationale: "For a plugin with <20 test files
and no compilation step, built-in `node:test` is simpler."

**Location for Nx library projects:** Follow Nx/Vitest convention:

```
libs/<lib-name>/
  src/

  src/

```

**Naming:**
- Plugin script tests: `<source-file-name>.test.mjs` (same name + `.test.`)
- Nx library tests: `<source-file-name>.spec.ts` (`.spec.ts` convention)

## Test Structure (node:test pattern)

```javascript
import { describe, it, before, after } from 'node:test';
import assert from 'node:assert/strict';

describe('WorkspaceIndexer', () => {
  let tempDir;

  before(async () => {
    // Setup: create temp directory with minimal nx.json fixture
    tempDir = await createTempWorkspace();
  });

  after(async () => {
    // Teardown: remove temp directory
    await removeTempWorkspace(tempDir);
  });

  describe('buildWorkspaceIndex', () => {
    it('returns projects from nx show projects output', async () => {
      const index = await buildWorkspaceIndex(tempDir, DEFAULT_CONFIG);

      assert.ok(index.projects);
      assert.strictEqual(typeof index.projects, 'object');
    });

    it('includes dependency edges from project graph', async () => {
      const index = await buildWorkspaceIndex(tempDir, DEFAULT_CONFIG);

      assert.ok(index.deps);
      assert.ok(Array.isArray(Object.values(index.deps)[0]));
    });

    it('throws when workspace root is not an Nx workspace', async () => {
      await assert.rejects(
        () => buildWorkspaceIndex('/tmp/not-a-workspace', DEFAULT_CONFIG),
        /\[ERROR\] No projects found/,
      );
    });
  });
});
```

**Patterns:**
- Setup: `before()` for async setup, `beforeEach()` for per-test reset
- Teardown: `after()` for async cleanup, `afterEach()` for per-test cleanup
- Assertions: `assert.strictEqual`, `assert.deepStrictEqual`, `assert.ok`,
  `assert.rejects`, `assert.throws` from `node:assert/strict`
- Async tests: `async` test functions, `await assert.rejects()` for error cases

## Mocking (node:test)

**Framework:** `node:test` built-in mock API (`mock.fn()`, `mock.method()`,
`mock.module()`).

**Patterns:**

```javascript
import { mock } from 'node:test';
import assert from 'node:assert/strict';

it('calls nx show projects with correct arguments', () => {
  // Mock execSync to avoid actual Nx CLI calls in unit tests
  const execSyncMock = mock.fn(() => JSON.stringify(['proj-a', 'proj-b']));
  mock.method(childProcess, 'execSync', execSyncMock);

  const projects = listProjects('/workspace');

  assert.strictEqual(execSyncMock.mock.calls.length, 1);
  assert.ok(execSyncMock.mock.calls[0].arguments[0].includes('show projects'));
});
```

**Module mocking for file system:**

```javascript
import { mock } from 'node:test';

it('reads workspace index from disk', async (t) => {
  // Use t.mock for test-scoped mocks (auto-restored after test)
  t.mock.method(fs, 'readFileSync', () => JSON.stringify(FIXTURE_INDEX));

  const index = loadWorkspaceIndex('/workspace/.nx/workspace-index.json');

  assert.strictEqual(index.version, 1);
});
```

**What to mock:**
- `node:child_process` `execSync`/`spawnSync` calls (Nx CLI via `execSync`, git grep via `spawnSync` with `shell: false`) -- too slow and
  require real workspaces in unit tests
- `node:fs` `readFileSync`/`writeFileSync` -- when testing logic, not I/O
- Time-based functions (`Date.now`) -- for consistent timestamp assertions

**What NOT to mock:**
- The `node:vm` sandbox itself -- test with real VM execution to catch
  real sandbox behavior and edge cases
- The handle store Map operations -- test with real in-memory Map
- Config loading from real fixture files -- use temp directories instead of mocking fs

## Fixtures and Factories

**Test data:** Use minimal fixture files representing real Nx CLI output formats.
Store fixtures under `plugins/lz-nx.rlm/scripts/__fixtures__/`:

```javascript
// __fixtures__/nx-show-projects.json -- minimal nx show projects output
export const NX_SHOW_PROJECTS_FIXTURE = ['proj-a', 'proj-b', 'proj-a-e2e'];

// __fixtures__/workspace-index.json -- minimal workspace index
export const WORKSPACE_INDEX_FIXTURE = {
  version: 1,
  generated: '2026-03-03T00:00:00.000Z',
  root: '/workspace',
  projects: {
    'proj-a': { name: 'proj-a', root: 'libs/proj-a', type: 'lib', tags: [], targets: ['build', 'test'] },
    'proj-b': { name: 'proj-b', root: 'apps/proj-b', type: 'app', tags: [], targets: ['build', 'serve'] },
  },
  deps: { 'proj-a': [], 'proj-b': ['proj-a'] },
  reverseDeps: { 'proj-a': ['proj-b'], 'proj-b': [] },
  aliases: { '@proj-a': 'libs/proj-a/src/index.ts' },
  stats: { projectCount: 2, fileCount: 0 },
};
```

**Factory pattern for REPL sandbox tests:**

```javascript
function createSandbox(overrides = {}) {
  return new ReplSandbox({
    workspaceIndex: WORKSPACE_INDEX_FIXTURE,
    config: { ...DEFAULT_CONFIG, ...overrides },
  });
}
```

**Location:**
- Fixtures: `plugins/lz-nx.rlm/scripts/__fixtures__/`
- Factory helpers: inline in the test file unless shared across multiple test files

## Coverage

**Requirements:** No enforced coverage thresholds configured in `nx.json`.
Per `.planning/research/STACK.md`, coverage is opt-in during development.

**View coverage:**
```bash
# node:test experimental coverage
node --test --experimental-test-coverage plugins/lz-nx.rlm/scripts/**/*.test.mjs

# Nx coverage (for Vitest projects)
pnpm nx test <project-name> --coverage
```

**Coverage priorities** (by component risk level, from `.planning/research/PITFALLS.md`):

| Component | Risk | Priority |
|-----------|------|----------|
| `repl-sandbox.mjs` | HIGH (VM escape, state corruption) | Must test sandbox escape vectors |
| Execution loop | HIGH (infinite loops, guardrails) | Must test all four guardrails |
| `workspace-indexer.mjs` | HIGH (Nx daemon timeouts) | Must test timeout handling |
| `handle-store.mjs` | MEDIUM (memory leaks) | Test eviction logic |
| `nx-runner.mjs` | LOW (allowlist logic) | Test allowlist enforcement |
| `path-resolver.mjs` | LOW (bidirectional logic) | Test round-trip accuracy |
| `rlm-config.mjs` | LOW (config merging) | Test default override |

## Test Types

**Unit Tests:**
- Scope: Individual script functions in isolation
- Mocking: Mock all I/O (`execSync`, `readFileSync`)
- Location: Co-located `.test.mjs` files
- Framework: `node:test` + `node:assert/strict`

**Integration Tests:**
- Scope: Full script execution with real file I/O against fixture workspaces
- Mocking: No mocking -- use temp directories with real fixture `nx.json`/`project.json`
- Location: `plugins/lz-nx.rlm/scripts/__tests__/` (separate subdirectory)
- Framework: `node:test` + `node:assert/strict`

**Sandbox Security Tests (critical):**
- Scope: Known VM escape payloads against `repl-sandbox.mjs`
- Purpose: Verify `codeGeneration: { strings: false, wasm: false }` blocks eval attacks
- Location: `plugins/lz-nx.rlm/scripts/repl-sandbox.security.test.mjs`
- Run on every change to `repl-sandbox.mjs` per `.planning/research/PITFALLS.md`

**E2E Tests:**
- Framework: Not configured (manual verification for plugin invocation from Claude Code)
- Verification checklist from `AGENTS.md`:
  1. Scripts execute correctly on the current platform
  2. Hook configurations are valid JSON
  3. Markdown files render correctly
  4. Commands work with expected arguments

## Common Patterns

**Async testing with node:test:**
```javascript
it('resolves path alias to filesystem path', async () => {
  const result = await resolveAlias('@proj-a', WORKSPACE_INDEX_FIXTURE);

  assert.strictEqual(result, 'libs/proj-a/src/index.ts');
});
```

**Error testing:**
```javascript
it('rejects disallowed Nx commands', async () => {
  const runner = new NxRunner({ allowlist: ['show', 'graph'] });

  await assert.rejects(
    () => runner.run('nx build my-app'),
    (err) => {
      assert.ok(err.message.includes('[ERROR] Command not in allowlist'));

      return true;
    },
  );
});
```

**RLM guardrail testing:**
```javascript
it('halts execution after maxConsecutiveErrors', async () => {
  const sandbox = createSandbox({ maxConsecutiveErrors: 2 });
  // Code that always throws
  const failingCode = 'throw new Error("deliberate failure");';

  const result1 = await sandbox.execute(failingCode);
  const result2 = await sandbox.execute(failingCode);
  const result3 = await sandbox.execute(failingCode);

  assert.ok(result3.halted);
  assert.ok(result3.error.includes('[ERROR] Aborted after 2 consecutive errors'));
});
```

**Sandbox variable persistence across iterations:**
```javascript
it('persists globalThis assignments across REPL turns', async () => {
  const sandbox = createSandbox();

  // Iteration 1: assign variable
  await sandbox.execute('const greeting = "hello"');

  // Iteration 2: read variable (should persist via globalThis transformation)
  const result = await sandbox.execute('FINAL(greeting)');

  assert.strictEqual(sandbox.getFinalAnswer(), 'hello');
});
```

**Platform-specific test skipping:**
```javascript
it('normalizes Windows backslash paths to forward slashes', {
  skip: process.platform !== 'win32' ? 'Windows-only test' : false,
}, () => {
  const normalized = normalizePath('plugins\\lz-nx.rlm\\scripts');

  assert.strictEqual(normalized, 'plugins/lz-nx.rlm/scripts');
});
```

## Nx Target Configuration

The `nx.json` configures caching for the `@nx/vitest:test` target:

```json
{
  "targetDefaults": {
    "@nx/vitest:test": {
      "cache": true,
      "inputs": ["default", "^production"]
    }
  }
}
```

The `production` named input excludes test files from production build caching:
```json
{
  "production": [
    "default",
    "!{projectRoot}/**/?(*.)+(spec|test).[jt]s?(x)?(.snap)",
    "!{projectRoot}/tsconfig.spec.json",
    "!{projectRoot}/src/test-setup.[jt]s"
  ]
}
```

## Verification Checklist (from AGENTS.md)

Before considering any plugin modification complete, verify:

1. Scripts execute correctly on the current platform (Windows + Git Bash)
2. Hook configurations are valid JSON (`JSON.parse` to verify)
3. Markdown files render correctly (preview in editor)
4. Commands work with expected arguments (manual invocation test)
