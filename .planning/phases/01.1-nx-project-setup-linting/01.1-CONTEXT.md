# Phase 1.1: Nx Project Setup + Linting - Context

**Gathered:** 2026-03-05
**Status:** Ready for planning

<domain>
## Phase Boundary

Plugin scripts and tests become discoverable Nx projects with inferred typecheck and lint targets, using recommended Nx ESLint rules. All existing code passes both targets with no suppression comments. Tests convert from .mjs to TypeScript. Formatting is normalized via `nx format --all` before lint rules are applied.

<decisions>
## Implementation Decisions

### JSDoc type checking (plugin scripts project only)
- `checkJs: true` + `allowJs: true` for plugin scripts (.mjs files checked via JSDoc annotations)
- `noEmit: true` -- validate only, no .d.ts declaration output
- Fix all type errors in existing .mjs scripts (no `@ts-ignore` or `@ts-expect-error` suppression)
- Test project does NOT need `checkJs` -- tests are .ts files checked natively by TypeScript
- Both projects get strict type checking consistent with `tsconfig.base.json` settings

### Test file format
- Convert all existing .mjs test files to TypeScript (.ts)
- Tests get native TypeScript type checking (no JSDoc ceremony)
- Vitest supports .ts natively via esbuild
- All tests in a project must be the same format (.ts, not a mix of .mjs and .ts)

### ESLint rule composition
- Full rule layers: eslint recommended + typescript-eslint recommended + Nx flat configs
- typescript-eslint rules scoped to `.ts` files ONLY (test project), not applied to .mjs files
- eslint-plugin-import for import ordering enforcement:
  - Groups: builtin, external, internal, parent, sibling
  - `node:*` pattern grouped as builtin (before position)
  - No newlines between import groups
- eslint-config-prettier as last config entry (disables ESLint rules that conflict with Prettier)
- Align all dependency choices (typescript-eslint, eslint-config-prettier, eslint-plugin-import) with Nx defaults for generated workspaces and projects

### Formatting
- Add `format` and `format:check` scripts to `package.json` using `nx format` and `nx format:check`
- First implementation step: run `nx format --all` to normalize formatting of all existing files before applying lint rules

### Module boundary enforcement
- Enable `@nx/enforce-module-boundaries` with tag-based constraints
- Tag scheme: `type:plugin` and `type:test` (not `scope:` prefix)
- Constraint: `type:plugin` can only depend on `type:plugin`; `type:test` can depend on `type:plugin` and `type:test`
- `enforceBuildableLibDependency: false` (plugin scripts are not buildable libraries)
- Keep existing `#rlm` tsconfig path alias registered in `tsconfig.base.json` for test-to-plugin imports

### Project structure (REQUIRES RESEARCH)
- Goal: Keep `project.json`, `eslint.config.mjs`, `vitest.config.*`, `tsconfig.json`, and test files OUTSIDE `plugins/<name>/` to maintain clean plugin distribution
- Research paths (in priority order):
  1. Shadow `projects/` directory pattern -- per-plugin subdirectory (e.g., `projects/lz-nx-rlm/`) containing project.json, eslint config, vitest config, and tests, with `sourceRoot` pointing to `plugins/lz-nx.rlm/scripts/`
  2. `projectRoot` / `sourceRoot` override in `project.json` -- can Nx decouple the config location from the source location?
  3. Custom Nx plugin that augments the project graph -- programmatically associate `plugins/<name>/` source with a project defined elsewhere
- Fallback: Decide after research completes. No premature commitment to a specific approach.
- Pattern must scale to future plugins in `plugins/<another-plugin>/`

### Claude's Discretion
- Exact eslint-plugin-import configuration details beyond the decided groups
- `allow` list exceptions in enforce-module-boundaries (handle as needed during implementation)
- tsconfig.json file structure and references for each project
- How to handle ESLint flat config cascading if shadow projects approach works
- Vitest config location decision (follows from project structure research)

</decisions>
