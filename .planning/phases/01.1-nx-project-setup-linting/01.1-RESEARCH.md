# Phase 1.1: Nx Project Setup + Linting - Research

**Researched:** 2026-03-05
**Domain:** Nx project configuration, ESLint flat config, TypeScript JSDoc type checking
**Confidence:** HIGH

## Summary

This phase requires creating two Nx projects (`lz-nx-rlm` for plugin scripts, `lz-nx-rlm-test` for tests) with inferred typecheck and lint targets, using Nx's recommended ESLint flat config with module boundary enforcement. The critical research question -- whether project config can live in a shadow `projects/` directory separate from source -- was answered definitively: **it cannot**. Nx's inferred target plugins (`@nx/js/typescript` and `@nx/eslint/plugin`) derive the project root from the `project.json` location and track files within `{projectRoot}/**/*` for cache invalidation. Source files outside this tree are invisible to both plugins.

The recommended approach is to place `project.json` directly in `plugins/lz-nx.rlm/` (making it the project root). The Nx TypeScript plugin will infer a `typecheck` target from a `tsconfig.json` in that directory. The Nx ESLint plugin will infer a `lint` target when a root-level `eslint.config.mjs` exists, cascading to all projects. This approach requires accepting that `project.json` and `tsconfig.json` ship with the plugin distribution -- these files are small, inert, and harmless to end users who install via repo clone.

A second critical finding: the Nx TypeScript plugin (v22.5.1) disables the `typecheck` target when `noEmit: true` is detected in tsconfig, despite TypeScript 5.9 supporting `tsc --build --noEmit`. The workaround is to use `emitDeclarationOnly: true` (inherited from `tsconfig.base.json`) and direct output to a gitignored `out-tsc/` directory, OR to override the inferred target with a custom `tsc -p tsconfig.json --noEmit` command that bypasses `--build` mode.

**Primary recommendation:** Place `project.json` and `tsconfig.json` in `plugins/lz-nx.rlm/`, use a single root `eslint.config.mjs` for lint targets, override the typecheck target in `project.json` to use `tsc -p tsconfig.json --noEmit` (avoids the Nx plugin's noEmit guard), and keep test infrastructure at `tests/lz-nx.rlm/` as established.

## Architecture Patterns

### Critical Finding: Shadow Projects Directory Does NOT Work

**Confidence: HIGH** -- Verified by reading Nx plugin source code directly.

The user's preferred approach (option 1 from CONTEXT.md) was to place `project.json` at `projects/lz-nx-rlm/` with `sourceRoot` pointing to `plugins/lz-nx.rlm/scripts/`. This approach **fails** for three independent reasons:

1. **ESLint plugin file discovery** (`@nx/eslint/plugin` v22.5.1, line 159): The `getRootForDirectory()` function walks UP from each lintable file's directory to find its owning project root. Files at `plugins/lz-nx.rlm/scripts/*.mjs` walk up `plugins/lz-nx.rlm/scripts/` -> `plugins/lz-nx.rlm/` -> `plugins/` -> `.` and never reach `projects/lz-nx-rlm/`. Those files would NOT be associated with the project for linting.

2. **Nx cache invalidation** (`nx.json` namedInputs): The `default` input is `{projectRoot}/**/*`. If `projectRoot = projects/lz-nx-rlm/`, changes to `plugins/lz-nx.rlm/scripts/*.mjs` would NOT invalidate the cache for typecheck or lint targets.

3. **TypeScript plugin project detection** (`@nx/js/typescript` v22.5.1, line 155): `projectRoot = dirname(configFilePath)` -- the project root is derived from `tsconfig.json` location. While `tsc` itself can `include` files from outside via relative paths, Nx's file tracking breaks.

**Option 2 (`sourceRoot` override):** `sourceRoot` is a metadata field used by some executors (e.g., asset copying). It does NOT change how Nx discovers projects, tracks inputs, or runs inferred targets. Setting `sourceRoot` to a different path does not solve any of the three problems above.

**Option 3 (custom Nx plugin):** A custom `createNodesV2` plugin could programmatically create projects with arbitrary roots and inputs. This is technically possible but extreme overkill for this use case -- it requires maintaining custom plugin code and fighting against the standard toolchain.

### Distribution Impact Assessment

Files that would now ship inside `plugins/lz-nx.rlm/`:

- `project.json` -- 3-5 lines, inert in other Nx workspaces
- `tsconfig.json` -- 10-15 lines, inert without the workspace's `tsconfig.base.json`

These files are harmless to end users. They are small, clearly workspace-tooling artifacts, and have no effect outside this specific Nx workspace. The AGENTS.md rule "Test files, test fixtures, and test runner configs MUST NOT live inside plugins/<name>/" applies specifically to test artifacts that would pollute the plugin. Config files like `project.json` and `tsconfig.json` are not test artifacts.

Note: `eslint.config.mjs` lives at the WORKSPACE ROOT (not inside `plugins/lz-nx.rlm/`), so it does NOT ship with the plugin.

### Pattern 1: Root ESLint Config Cascading to All Projects

**What:** A single `eslint.config.mjs` at the workspace root provides lint configuration for all projects. The `@nx/eslint/plugin` groups each project root under its nearest parent ESLint config. With only a root config, all projects get lint targets automatically.
**When to use:** Always in Nx workspaces unless projects need fundamentally different ESLint configurations.
**Source:** `@nx/eslint/plugin` source code, `groupProjectRootsByEslintRoots()` function (line 137-148).

**Example:**

```javascript
// eslint.config.mjs (workspace root)
import nx from '@nx/eslint-plugin';
import eslintConfigPrettier from 'eslint-config-prettier';

export default [
  ...nx.configs['flat/base'],
  ...nx.configs['flat/typescript'],
  ...nx.configs['flat/javascript'],
  {
    ignores: ['**/dist', '**/out-tsc', '**/node_modules'],
  },
  {
    files: ['**/*.ts', '**/*.tsx', '**/*.js', '**/*.jsx', '**/*.mjs'],
    rules: {
      '@nx/enforce-module-boundaries': [
        'error',
        {
          enforceBuildableLibDependency: false,
          allow: ['^.*/eslint(\\.base)?\\.config\\.[cm]?[jt]s$'],
          depConstraints: [
            {
              sourceTag: 'type:plugin',
              onlyDependOnLibsWithTags: ['type:plugin'],
            },
            {
              sourceTag: 'type:test',
              onlyDependOnLibsWithTags: ['type:plugin', 'type:test'],
            },
          ],
        },
      ],
    },
  },
  eslintConfigPrettier,
];
```

### Pattern 2: TypeScript Plugin Typecheck Target Override

**What:** The Nx TypeScript plugin (v22.5.1) disables the inferred `typecheck` target when `noEmit: true` is in tsconfig. Override the target in `project.json` with a custom command.
**When to use:** When a project needs `noEmit: true` (no output files) but still wants type checking.
**Source:** `@nx/js/src/plugins/typescript/plugin.js` lines 250-254.

**Example:**

```jsonc
// plugins/lz-nx.rlm/project.json
{
  "name": "lz-nx-rlm",
  "tags": ["type:plugin"],
  "targets": {
    "typecheck": {
      "command": "tsc -p {projectRoot}/tsconfig.json --noEmit",
      "cache": true,
      "inputs": [
        "{projectRoot}/**/*.mjs",
        "{projectRoot}/tsconfig.json",
        "{workspaceRoot}/tsconfig.base.json",
      ],
      "outputs": [],
    },
  },
}
```

**Alternative approach (no override needed):** Do NOT set `noEmit: true` in tsconfig.json. Instead, let the Nx plugin use `tsc --build --emitDeclarationOnly`. The generated `.d.mts` files go to an `outDir` (e.g., `../../out-tsc/plugins/lz-nx.rlm/`), which is gitignored. Type checking still happens; the only side effect is .d.ts files in a temp directory.

### Pattern 3: TSConfig Structure for JSDoc-Annotated .mjs Files

**What:** TypeScript config that enables JSDoc type checking on .mjs files while respecting the workspace's composite project references setup.
**When to use:** For the `lz-nx-rlm` plugin scripts project.

**Example:**

```jsonc
// plugins/lz-nx.rlm/tsconfig.json
{
  "extends": "../../tsconfig.base.json",
  "compilerOptions": {
    "allowJs": true,
    "checkJs": true,
    "noEmit": true,
    "composite": false,
    "declaration": false,
    "declarationMap": false,
    "emitDeclarationOnly": false,
    "rootDir": "scripts",
  },
  "include": ["scripts/**/*.mjs"],
  "exclude": ["node_modules"],
}
```

**Critical notes:**

- `composite: false` overrides `tsconfig.base.json` -- required because `composite` and `noEmit` are incompatible
- `declaration: false`, `declarationMap: false`, `emitDeclarationOnly: false` -- override base to prevent any output
- `allowJs: true` + `checkJs: true` enables JSDoc type checking on `.mjs` files
- `module: "nodenext"` and `moduleResolution: "nodenext"` (from base) are required for `.mjs` extension recognition
- Because `composite: false` and `noEmit: true` disable the inferred typecheck target, an explicit target override is needed in `project.json`

### Pattern 4: TSConfig Structure for TypeScript Test Files

**What:** TypeScript config for the test project with native .ts type checking.
**When to use:** For the `lz-nx-rlm-test` test project.

**Example:**

```jsonc
// tests/lz-nx.rlm/tsconfig.json
{
  "extends": "../../tsconfig.base.json",
  "compilerOptions": {
    "outDir": "../../out-tsc/tests/lz-nx.rlm",
    "types": ["vitest/globals", "node"],
    "paths": {
      "#rlm/*": ["../../plugins/lz-nx.rlm/scripts/*"],
    },
  },
  "include": ["**/*.ts"],
  "exclude": ["node_modules"],
  "references": [{ "path": "../../plugins/lz-nx.rlm" }],
}
```

**Notes:**

- `paths` re-declares the `#rlm` alias at project level (TypeScript project references require this)
- `references` establishes a dependency on the plugin scripts project
- The `outDir` keeps build artifacts out of the test directory

### Anti-Patterns to Avoid

- **Shadow directory for project config:** Placing `project.json` in a different directory tree than source files breaks Nx inferred targets, cache invalidation, and file discovery.
- **`noEmit: true` without target override:** The Nx TypeScript plugin silently disables the typecheck target. The echo message appears at runtime, not at configuration time, making it hard to debug.
- **Per-project ESLint configs when root suffices:** Adding `eslint.config.mjs` to each project directory creates maintenance burden. A single root config with file-pattern scoping handles most cases.
- **Mixing `composite: true` with `noEmit: true`:** TypeScript does not allow this combination. Must set `composite: false` when using `noEmit: true`.

## Common Pitfalls

### Pitfall 1: noEmit Silently Disables Typecheck Target

**What goes wrong:** The `typecheck` target runs but just echoes a warning message instead of type-checking.
**Why it happens:** `@nx/js/typescript` plugin (v22.5.1) checks `tsConfig.options.noEmit` and replaces the command with an echo when true (line 250-254 of plugin.js).
**How to avoid:** Either (a) override the typecheck target in `project.json` with a custom `tsc` command, or (b) use `emitDeclarationOnly: true` with an `outDir` instead of `noEmit: true`.
**Warning signs:** `nx typecheck lz-nx-rlm` succeeds instantly with output like "The 'typecheck' target is disabled..."

### Pitfall 2: composite + noEmit Incompatibility

**What goes wrong:** TypeScript error TS5053: "Option 'noEmit' cannot be specified with option 'composite'."
**Why it happens:** `tsconfig.base.json` sets `composite: true`. Project tsconfig inheriting this plus adding `noEmit: true` creates a conflict.
**How to avoid:** Override `composite: false` in the project's tsconfig.json when using `noEmit: true`.
**Warning signs:** tsc compilation errors about incompatible options.

### Pitfall 3: .mjs Files Not Included in TypeScript Checking

**What goes wrong:** TypeScript silently skips `.mjs` files -- no errors reported even when JSDoc types are wrong.
**Why it happens:** TypeScript only recognizes `.mjs` when `moduleResolution` is `node16`, `nodenext`, or `bundler`. Also, `allowJs` must be `true` and the `include` glob must explicitly cover `**/*.mjs`.
**How to avoid:** Ensure `moduleResolution: "nodenext"` (from base), `allowJs: true`, `checkJs: true`, and `include: ["scripts/**/*.mjs"]`.
**Warning signs:** `tsc --noEmit` reports zero errors on a fresh project -- too good to be true.

### Pitfall 4: ESLint Flat Config File Pattern Differences

**What goes wrong:** Rules don't apply to expected files.
**Why it happens:** ESLint 9 flat config uses `files: ['**/*.ts']` (with `**/` prefix) not `files: ['*.ts']` (legacy overrides format). Nx's generated config uses the `**/*` prefix pattern.
**Warning signs:** ESLint passes when it should fail on TypeScript-specific rules.

### Pitfall 5: typescript-eslint Rules Applied to .mjs Files

**What goes wrong:** Lint errors like "Parsing error: Unexpected token" on .mjs files.
**Why it happens:** typescript-eslint parser can't parse plain JavaScript .mjs files without `allowJs` in the parser config, or rules designed for TypeScript syntax fail on plain JS.
**How to avoid:** Scope typescript-eslint recommended rules to `files: ['**/*.ts', '**/*.tsx']` only. Do NOT apply them to `**/*.mjs` files.
**Warning signs:** ESLint failures in .mjs files mentioning TypeScript-specific syntax.

### Pitfall 6: Test Conversion .mjs -> .ts Missing Vitest Patterns

**What goes wrong:** Vitest finds no tests after conversion.
**Why it happens:** Existing vitest.config.mjs has `include: ['**/*.test.mjs']`. After converting to .ts, this pattern no longer matches.
**How to avoid:** Update vitest.config include to `['**/*.test.ts']` when converting test files.
**Warning signs:** `nx test lz-nx-rlm-test` reports 0 test files found.

## Open Questions

1. **eslint-plugin-import vs eslint-plugin-import-x**
   - What we know: Nx does NOT include either by default. `eslint-plugin-import-x` has 16 vs 117 deps, native flat config, Rust-based resolver, ESLint team endorsement.
   - What's unclear: Whether `eslint-plugin-import` flat config support is fully stable. Whether import-x handles `node:*` prefix grouping identically.
   - Recommendation: Use `eslint-plugin-import-x` (modern, fewer deps, better flat config support). Falls under Claude's discretion per CONTEXT.md.

2. **tsconfig.base.json paths for #rlm alias**
   - What we know: CONTEXT.md says "Keep existing `#rlm` tsconfig path alias registered in `tsconfig.base.json`". Currently `tsconfig.base.json` does NOT have a `paths` entry -- the alias only exists in vitest.config.mjs resolve.alias.
   - What's unclear: Whether the user expects `#rlm` in tsconfig.base.json (for TypeScript resolution) or just in vitest config (for test bundling). TypeScript path aliases and Vitest resolve.alias serve different purposes.
   - Recommendation: Add `#rlm/*` to tsconfig.base.json paths AND keep vitest resolve.alias. Both are needed -- TypeScript for type checking across project references, Vitest for test module resolution.

3. **Root tsconfig.json references**
   - What we know: Currently `tsconfig.json` has empty `references: []`. Both projects need to be added. The `@nx/js:typescript-sync` generator can auto-manage this.
   - What's unclear: Whether `typescript-sync` handles the case where `composite: false` in one project's tsconfig. Non-composite projects may not be valid project references.
   - Recommendation: Since the plugin scripts project has `composite: false` (for noEmit compatibility), it cannot be a TypeScript project reference in the traditional sense. Only add the test project as a reference. The plugin project gets its types checked independently.
