    - Filter OUT entries where the key contains `*` (wildcard patterns) -- per user decision, these are ignored
    - For remaining entries: map key -> the FULL path array (preserve all elements, do NOT take only the first). TypeScript `paths` values are arrays representing fallback resolution order, and the user decision requires all paths be shown. For example, `"@myorg/shared-utils": ["libs/shared-utils/src/index.ts"]` stays as `{ "@myorg/shared-utils": ["libs/shared-utils/src/index.ts"] }`. An alias with multiple fallbacks like `"@myorg/utils": ["libs/utils/src/index.ts", "libs/utils-compat/src/index.ts"]` would be preserved as `{ "@myorg/utils": ["libs/utils/src/index.ts", "libs/utils-compat/src/index.ts"] }`.
    - Strip baseUrl prefix if present from each path in the arrays (paths are relative to baseUrl)
    - Return the alias->paths map object (values are string arrays, may be empty object)

  <action>
    **path-resolver.mjs:**

    `resolveAlias(input, pathAliases)`:
    - pathAliases values are string arrays (e.g., `{ "@myorg/shared-utils": ["libs/shared-utils/src/index.ts"] }`)
    - If input is empty/undefined, return `{ results: [], partial: false, error: 'Missing required argument' }`
    - **Step 1 - Exact alias match:** Check if `input` is a key in pathAliases. If found, return `{ results: [{ from: input, to: pathAliases[input], direction: 'alias->path' }], partial: false }`. The `to` field is the full string[] array of paths for that alias, preserving TypeScript fallback resolution order per user decision.
    - **Step 2 - Exact path match:** Iterate all entries in pathAliases. For each alias, check if `input` matches any element in its paths array. If found, return all matching entries as `{ from: alias, to: input, direction: 'path->alias' }`. Multiple matches possible (same path can appear in different aliases).
    - **Step 3 - Substring fallback (alias side):** Search alias keys for substring match with `input`. Collect matches as `{ from: alias, to: pathAliases[alias], direction: 'alias->path' }`.
    - **Step 4 - Substring fallback (path side):** If no alias-side substring matches, search path value arrays for any element containing `input` as substring. Collect as `{ from: alias, to: matchedPath, direction: 'path->alias' }`.
    - For substring results: truncate at 20, set `partial: true`, include which side matched in the results.
    - Return `{ results, partial }` where each result is `{ from, to, direction }`.

    The auto-detect direction logic per user decision: "check alias keys first (exact), then path values (exact), then substring fallback". This is exactly the step order above.

    For unscoped aliases (no `@` prefix): the lookup order handles them naturally -- they're just shorter keys checked against the same map.
