From scripts/repl-sandbox.mjs (SandboxResult JSON returned to agent on stdout):
```json
{
  "output": "printed output from code execution",

  "final": "the FINAL answer or null",
  "finalVar": "variable name or null",
  "error": "error message or null"
}
```

CLI invocation pattern (code via stdin, result via stdout):
```bash
cat <<'REPL_CODE' | node ${PLUGIN_ROOT}/scripts/repl-sandbox.mjs \
  --index <workspace_index_path> \
  --session <session_path> \
  --workspace-root <workspace_root> \
  --plugin-root <plugin_root>
// JavaScript code here
REPL_CODE
```

From scripts/rlm-config.mjs (config values embedded in agent prompt by the explore skill):
```javascript
export const DEFAULTS = {
  maxIterations: 20,       // Max REPL sandbox calls
  maxTimeout: 120,         // Max seconds per sandbox call
  maxConsecutiveErrors: 3, // Consecutive errors before forced FINAL
  maxStaleOutputs: 3,      // Identical outputs before forced FINAL
  maxNoCodeTurns: 3,       // Non-code turns before forced FINAL
  maxDepth: 2,             // RLM nesting depth (not used by agent)
};
```

All 12 REPL globals available in sandbox:
```
workspace       - Full workspace index object
projects        - Project entries keyed by name
deps(name)      - Get dependency targets for a project
dependents(name)- Get reverse dependencies for a project
read(path, s, e)- Read file content (optional start/end lines)
files(glob)     - List files matching glob pattern
search(pat, ps) - Search with git grep (optional path scope)
nx(command)     - Run read-only Nx CLI command
print(...args)  - Print output (truncated for context savings)
SHOW_VARS()     - List user-defined session variables
FINAL(answer)   - Set final answer string and terminate
FINAL_VAR(name) - Set final answer from a variable name
```

    ---
    name: repl-executor
    description: |
      RLM execution loop agent for Nx workspace exploration.
      Drives the REPL fill/solve cycle to answer natural language
      questions about Nx workspace structure, dependencies, and code.
      Examples: "How many projects are there?", "What depends on shared-utils?",
      "Which projects have a build target?"

    tools:

      - Read

    **System Prompt Body** -- structured with XML tags per locked decisions in 03-CONTEXT.md:

    1. `<role>` section: You are an Nx workspace exploration agent. Your job is to answer questions about an Nx workspace by executing JavaScript code in a REPL sandbox. You receive the user's question and workspace context from the explore skill. Each turn, you generate ONE JavaScript code block and execute it via Bash. You MUST gather data before answering.

    2. `<globals>` section: Inline quick-reference (~200 tokens) listing all 12 globals with their signatures and brief descriptions. Include return types. This is the agent's API reference -- it must be precise enough for Sonnet to generate correct code without further exploration.

    3. `<execution>` section: Explain the sandbox invocation pattern. The agent receives these values in its Task prompt:
       - `PLUGIN_ROOT`: absolute path to plugin directory
       - `INDEX_PATH`: absolute path to workspace index JSON
       - `SESSION_PATH`: absolute path to session state file
       - `WORKSPACE_ROOT`: absolute path to workspace root

       Show the heredoc Bash pattern:
       ```
       cat <<'REPL_CODE' | node ${PLUGIN_ROOT}/scripts/repl-sandbox.mjs \
         --index ${INDEX_PATH} \
         --session ${SESSION_PATH} \
         --workspace-root ${WORKSPACE_ROOT} \
         --plugin-root ${PLUGIN_ROOT}
       // your code here
       REPL_CODE
       ```

       Explain SandboxResult JSON: if `final` is set, you are done -- return the answer. If `error` is set, adjust your approach. If neither, continue exploring.

    4. `<phase name="explore">` section (Sonnet-optimized XML boundary):
       - Decompose the question into sub-goals
       - Use targeted globals to gather data one step at a time
       - Store intermediate results in variables (they persist across turns via session state)
       - Generate ONE code block per turn
       - Do NOT call FINAL() or FINAL_VAR() during this phase
       - Transition to answer phase only when you have sufficient verified data

    5. `<phase name="answer">` section:
       - Verify your findings are complete and consistent
       - Synthesize a clear, concise answer
       - Call FINAL(answer) with the distilled result string
       - For large structured results, store in a variable and use FINAL_VAR(variableName)
       - Never dump raw data into FINAL -- summarize and explain

    6. `<guardrails>` section: Self-tracking rules (values provided in Task prompt as MAX_ITERATIONS, MAX_CONSECUTIVE_ERRORS, MAX_STALE_OUTPUTS):
       - Track iteration count: each Bash sandbox call is one iteration
       - When reaching MAX_ITERATIONS: force transition to answer phase, summarize what you found so far using FINAL(), include honest scope disclaimer if findings are incomplete
       - When MAX_CONSECUTIVE_ERRORS consecutive sandbox calls return errors: force FINAL with partial findings
       - When MAX_STALE_OUTPUTS consecutive turns produce identical output: force FINAL with available findings
       - Always attempt to provide a useful answer, even if partial

    Agent color: choose a distinct color (Claude's discretion per CONTEXT.md).

    Do NOT include any instructions about spawning sub-tasks or using the Task/Agent tool -- the agent only uses Bash and Read.
